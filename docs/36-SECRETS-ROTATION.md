# Uelfy Clinical — Secrets Rotation Policy & Runbook

> **Scope.** Inventory of all secrets used by the platform, recommended
> rotation frequency, step-by-step procedure to rotate each one, and
> post-rotation verification. Written for the founder/operator running
> production.
>
> **Audience.** Anyone with admin access to the Vercel project, the
> Supabase project, the Upstash account, and the GitHub repository.
> Currently: founder only.
>
> **Companion docs:**
> - `docs/20-SECURITY.md` — security posture overview
> - `docs/26-DEPLOYMENT-RUNBOOK.md` — env vars table and deploy procedure
> - `docs/27-INCIDENT-RESPONSE.md` — when a secret is suspected leaked
>
> **Status.** Sprint 2 task 2.5 — first version. Reflects the current
> set of secrets at the time of writing. Update this doc whenever a
> new secret is introduced or an existing one is retired.

## 1. Why rotate secrets

Even with no observed compromise, secrets get rotated periodically because:

1. **Insider risk.** A secret known to a former contributor (or to a
   compromised third-party tool) that was never rotated stays valid
   forever. Periodic rotation shrinks the blast radius of any past
   leak you might not have noticed.
2. **Audit posture.** Clinical procurement security questionnaires
   (ISO 27001, SOC 2, ENS in EU) ask "what is your secret rotation
   policy?". A documented schedule + evidence of execution is the
   canonical answer.
3. **Defence-in-depth.** If a secret leaks via a transient channel
   (build log, debug session, shoulder surfing), a 90-day rotation
   guarantees the leak is harmless after at most 90 days.

Rotation is NOT a substitute for incident response when a leak is
suspected. See `docs/27-INCIDENT-RESPONSE.md` for that path.

## 3. Rotation procedures

### 3.1 `SUPABASE_SERVICE_ROLE_KEY` — every 90 days

**Pre-flight check (5 min):**

1. Verify production is healthy before rotating:
   ```bash
   curl -s https://prevention2.vercel.app/api/v1/health | jq '.subsystems'
   ```
   All three subsystems must be `"ok"`. If any is degraded, fix that
   first — do not rotate during an incident.
2. Open Supabase dashboard:
   ```
   https://supabase.com/dashboard/project/<project-ref>/settings/api
   ```
3. Open Vercel dashboard:
   ```
   https://vercel.com/<org>/prevention2/settings/environment-variables
   ```

**Rotation (10 min):**

1. **Supabase**: in the API settings page, find the `service_role`
   key (not the `anon` key). Click **Generate new key** (or
   **Roll** depending on the Supabase UI version). Confirm —
   Supabase invalidates the old key and shows the new one ONCE.
2. **Copy the new key immediately** to a secure clipboard buffer.
3. **Vercel**: in the Environment Variables page, find
   `SUPABASE_SERVICE_ROLE_KEY`:
   - Click the row → **Edit**
   - Paste the new value
   - Confirm scope: ☑ Production ☑ Preview ☐ Development
   - Click **Save**
4. Vercel hot-reloads env vars on the NEXT request. No redeploy.
5. **Wait ~30 seconds** for in-flight requests to complete with the
   old key (they were already authenticated when the request started).
6. **Verify** (within 2 min — old key still valid for the in-flight
   window):
   ```bash
   curl -s https://prevention2.vercel.app/api/v1/health \
        | jq '.subsystems[] | select(.name=="supabase")'
   ```
   Must return `"status": "ok"` with a sub-second latencyMs. If it
   returns `"down"` with `"detail": "query_error"`, the new key was
   not saved correctly on Vercel — re-check step 3.

**Post-rotation (2 min):**

1. **Clear the clipboard** (paste anything else over the secret).
2. **Update the rotation log** in `docs/36-SECRETS-ROTATION.md` §6.
3. **Calendar reminder**: set a reminder 80 days from today to begin
   the next rotation cycle (gives a 10-day window to handle
   scheduling).

### 3.2 `CRON_SIGNING_SECRET` — every 180 days

This secret authorises calls to `api/v1/internal/retention.ts` and
`api/v1/internal/anonymize.ts`. Only Vercel Cron + your operator
manual triggers should know it.

**Rotation (10 min):**

1. **Generate a new strong secret** (Mac):
   ```bash
   openssl rand -base64 48
   ```
   Copy the output (~64 chars). It MUST stay confidential.
2. **Vercel**: edit `CRON_SIGNING_SECRET` env var, paste the new
   value, save (Production scope).
3. **Verify cron continues to run on schedule** (next retention cron
   fires at 03:00 UTC daily, anonymize at 04:00 UTC):
   - Open Vercel dashboard → Cron Jobs (or Functions tab → look for
     `internal/retention` invocation logs)
   - Check the next cron run shows status 200 (not 401 — would mean
     Vercel Cron isn't sending the new secret)
4. If cron runs return 401, the new env var didn't propagate to
   Vercel Cron. Force a redeploy (Deployments → latest → ⋯ →
   **Redeploy**).

**Post-rotation:**

1. Clear clipboard.
2. Update §6 log.
3. Calendar reminder for 170 days from today.

### 3.3 `UPSTASH_REDIS_REST_URL` + `UPSTASH_REDIS_REST_TOKEN` — every 90 days (token only)

**Rotation (10 min):**

1. **Upstash**: go to
   ```
   https://console.upstash.com/redis/<database-id>
   ```
   In the **Details** panel, find **REST API** → click **Reset
   Token**. Upstash invalidates the old token and shows the new one
   once.
2. The URL **does not change** unless you migrate the database to a
   different region — leave `UPSTASH_REDIS_REST_URL` as-is.
3. **Vercel**: edit `UPSTASH_REDIS_REST_TOKEN` env var, paste the
   new value, save (Production + Preview scope).
4. **Verify**:
   ```bash
   curl -s https://prevention2.vercel.app/api/v1/health \
        | jq '.subsystems[] | select(.name=="rate_limit_distributed")'
   ```
   Must return `"status": "ok"`. If `"degraded"` with
   `"detail": "falling_back_to_memory"`, the new token didn't reach
   the lambda — re-check step 3 and force a redeploy if needed.

**Post-rotation:**

1. Clear clipboard.
2. Update §6 log.
3. Calendar reminder for 80 days from today.

### 3.4 `DATABASE_URL_STAGING` (when staging is provisioned — Sprint 2 task 2.2)

**Pre-requisite:** Sprint 2 task 2.2 must be done first
(`docs/37-SUPABASE-STAGING-SETUP.md`).

**Rotation (10 min):**

1. **Supabase staging dashboard**:
   ```
   https://supabase.com/dashboard/project/<staging-ref>/settings/database
   ```
   Reset the database password (Database Settings → Reset password).
2. **Copy the new connection string** (URI format) immediately.
3. **GitHub**: go to
   ```
   https://github.com/luchino99/Prevention2/settings/secrets/actions
   ```
   Click `DATABASE_URL_STAGING` → **Update**. Paste the new value.
   Save.
4. **Verify**: trigger a manual workflow run (Actions tab → CI →
   Run workflow). The `Build gates` step must show `[run-rls-tests]
   OK` and `[check-rls-coverage] OK` (not SKIP).

**Post-rotation:**

1. Clear clipboard.
2. Update §6 log.
3. Calendar reminder for 80 days from today.

### 3.5 GitHub Personal Access Tokens (operator-side)

If you're using `gh` CLI or scripted GitHub API calls with a personal
access token (PAT):

1. Visit `https://github.com/settings/tokens`
2. Find the token in use → **Regenerate token** (or **Delete** + create
   new with same scopes)
3. Update wherever the token is configured (`gh auth login`, scripts,
   CI variables outside Prevention2)

PATs scoped only to `repo` for Prevention2 are lower risk than
classic admin tokens. Prefer fine-grained PATs over classic ones.

## 4. Emergency rotation (suspected leak)

If you suspect any secret has leaked (build log accidentally tweeted,
laptop stolen, ex-contributor goes rogue):

1. **Do NOT post anything publicly** about the suspected leak (would
   accelerate the attacker's response).
2. **Rotate IMMEDIATELY**, skipping the pre-flight check:
   - Supabase service-role: §3.1 steps 1-5, do all in <5 min
   - Cron secret: §3.2 steps 1-3
   - Upstash token: §3.3 steps 1-3
3. **Audit log review** post-rotation:
   ```sql
   -- run via Supabase SQL editor on production
   SELECT *
     FROM audit_events
    WHERE created_at > NOW() - INTERVAL '7 days'
      AND (
        action LIKE '%service_role%'
        OR action IN ('patient.export', 'admin.audit.read', 'dsr.fulfill')
      )
    ORDER BY created_at DESC
    LIMIT 100;
   ```
   Look for unexpected actor IDs, IPs, or burst patterns.
4. **Open an incident** per `docs/27-INCIDENT-RESPONSE.md`. Even if
   nothing was exfiltrated, the incident process documents the event
   for compliance.

## 5. Verification after EVERY rotation

The post-rotation verification curl is the SAME for all secrets that
affect Vercel runtime:

```bash
PROD_URL=https://prevention2.vercel.app
curl -sS "$PROD_URL/api/v1/health" | jq '{
  status: .status,
  subsystems: .subsystems
}'
```

Expected output (all green):

```json
{
  "status": "ok",
  "subsystems": [
    {"name": "supabase",                 "status": "ok",  "latencyMs": <100>},
    {"name": "rate_limit_distributed",   "status": "ok"},
    {"name": "mfa_enforcement",          "status": "ok"}
  ]
}
```

If `status` is `"degraded"` after rotation, the new secret value did
not reach the lambda — re-check the Vercel env var, optionally force
a redeploy.

The CI smoke-prod job (Sprint 2 tasks 2.3, 2.4) runs the same checks
on every push and will alarm automatically if a future deploy goes
out with a broken secret.

## 7. Calendar reminders to set today

To make rotation actually happen, set these recurring reminders in
your calendar of choice (Google Calendar, Apple Calendar, etc.):

- **Every 90 days**: "Rotate SUPABASE_SERVICE_ROLE_KEY (docs/36)"
- **Every 90 days**: "Rotate UPSTASH_REDIS_REST_TOKEN (docs/36)"
- **Every 180 days**: "Rotate CRON_SIGNING_SECRET (docs/36)"
- **Annually (e.g. 1 January)**: "Review docs/36-SECRETS-ROTATION.md
  for new secrets that need to be added to the policy"

A rotation policy that nobody remembers to execute is worth nothing.
The reminders are the actual control.
