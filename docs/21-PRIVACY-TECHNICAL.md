# Uelfy Clinical — Privacy by Design (Technical View)

> **Scope.** Technical implementation of GDPR's privacy-by-design and
> privacy-by-default principles (Art.25). Companion to
> `20-SECURITY.md` (security architecture), `22-GDPR-READINESS.md`
> (article-by-article readiness), and `14-DELETION-POLICY.md` (retention).
>
> **Audience.** Engineers, DPOs, tenant privacy teams.
>
> **What this is not.** A legal opinion. The legal interpretation of
> "necessity" or "purpose compatibility" is the data controller's
> responsibility — typically the tenant clinic. Items marked `EXT-LEGAL`
> require sign-off by the controller's counsel.

---

## 1. Roles under GDPR

| Actor | GDPR role |
|---|---|
| Tenant clinic / hospital | **Data controller** for its patients. |
| Uelfy (the platform operator) | **Data processor**, under DPA with each tenant. |
| Patient (data subject) | Data subject. |
| Clinician (logged-in user of a tenant) | Acts on behalf of the controller. |
| Sub-processors (Supabase, Vercel, etc.) | **Sub-processors**, listed in the per-tenant DPA. |

The DPA is an `EXT-LEGAL` deliverable — counsel-drafted, not generated by
this codebase.

## 2. Categories of personal data processed

| Category | Examples | Lawful basis (typical) | Notes |
|---|---|---|---|
| Identifying PHI | name, DoB, sex, contact info | Art.6(1)(b) contract + Art.9(2)(h) healthcare | Stored in `patients` |
| Clinical PHI | lab values, vitals, medications, diagnoses | Art.9(2)(h) healthcare | Stored in `assessment_measurements`, `patient_clinical_profiles`, `clinical_input_snapshot` |
| Computed clinical data | scores, risk levels, alerts, follow-up plans | Art.9(2)(h) | Stored in `score_results`, `risk_profiles`, `alerts`, `followup_plans`, `due_items` |
| Lifestyle adherence | PREDIMED responses, activity minutes | Art.9(2)(h) (where part of care) or Art.6(1)(a) consent | Stored in `nutrition_snapshots`, `activity_snapshots` |
| Account data (clinicians) | name, email, role, MFA factor | Art.6(1)(b) contract | Stored in `users` |
| Telemetry | actor_user_id, action, hashed IP, truncated UA | Art.6(1)(f) legitimate interest | Stored in `audit_events` |
| Consent records | consent type, granted, policy_version, hashed IP | Art.7(1) accountability | Stored in `consent_records` |
| DSR records | kind, subject id, sla_deadline, fulfilled_at | Art.12(3) record-keeping | Stored in `data_subject_requests` |

The `EXT-LEGAL` choice of lawful basis per data category is the
controller's. The default labels above represent the **typical** healthcare
configuration.

## 3. Data minimisation

Implemented as code, not policy:

- The clinical engine receives a **typed snapshot** (`AssessmentInput`)
  composed of validated Zod schemas — extra fields sent by the client are
  **stripped** by `.strict()` schemas, not silently stored.
- The audit logger sanitises metadata to an allowlist (no PHI in audit
  rows; see `sanitizeMetadata` in `audit-logger.ts`). Strings are truncated
  to 256 chars, arrays to 20 elements, objects dropped.
- `recordFailedLogin` captures only the **email domain** (`split('@')[1]`),
  never the full email, when an unauthenticated user fails to log in.
- The `patients` row stores only the data the tenant explicitly entered.
  There is no enrichment, no third-party API call to add fields the
  clinician didn't ask for.
- Patient export (Art.15/20) returns the actual stored bytes only — it
  does not include re-derived metrics. Audit IPs are omitted from the
  export envelope (kept in `audit_events` server-side for accountability,
  but not handed back to the data subject).

## 4. Purpose limitation

- Each table is single-purpose by schema. There is no "miscellaneous JSON"
  free-text column outside of:
  - `patients.notes` (clinician-entered free text)
  - `assessments.notes` (per-assessment free text)
  - `clinical_input_snapshot` (deterministic snapshot of inputs for
    reproducibility)
  - `consent_records.purpose` (Art.5(1)(b) requires per-purpose
    declaration)
- Audit `metadata_json` is allowlist-sanitised — it cannot accidentally
  carry PHI from a debugging change.
- `audit_events.action` is a fixed enum (`AuditAction` in
  `audit-logger.ts`). New actions go through a typed addition, not free
  strings.

## 5. Storage limitation (retention)

The full retention matrix lives in `14-DELETION-POLICY.md`. Summary of the
defaults (each is configurable per tenant `EXT-LEGAL`):

| Data class | Default | Mechanism |
|---|---|---|
| Active patients | While clinically active | Soft-delete on revoke / discharge |
| Soft-deleted patients | 30-day grace | `fn_anonymize_patient()` after grace |
| Assessments | 10 years (medical record default) | Anonymisation, not deletion (clinical lineage retained) |
| Audit events | 7 years (Art.30 record + EU healthcare ledger conventions) | `fn_retention_prune()` |
| DSR records | 10 years (Art.12(3) record-keeping) | Manual archive |
| Notification queue | 90 days | `fn_retention_prune()` |
| Anonymised patients | Indefinite (no longer personal data) | Stripped of identifiers; clinical scalars retained |

Cron schedule:

| Cron path | Schedule | Function |
|---|---|---|
| `/api/v1/internal/anonymize` | `0 4 * * *` | Anonymise soft-deleted patients past grace |
| `/api/v1/internal/retention` | `0 3 * * *` | Prune expired audit / notification / alert rows |

Both cron handlers require the `CRON_SIGNING_SECRET` bearer header
(constant-time compare, B-04).

## 6. Subject rights — implementation map

| Right | Article | Implementation | Endpoint |
|---|---|---|---|
| Information | Art.13/14 | Privacy notice in onboarding flow + `/legal/privacy` page | (Frontend) |
| Access | Art.15 | JSON envelope export with full record | `GET /api/v1/patients/[id]/export` |
| Rectification | Art.16 | Standard PATCH on patient + assessment | `PATCH /api/v1/patients/[id]` |
| Erasure | Art.17 | Soft-delete + 30-day grace + `fn_anonymize_patient()` | `DELETE /api/v1/patients/[id]` then automatic |
| Restriction | Art.18 | Soft-delete (no further processing); explicit `restriction` DSR kind in ledger | DSR endpoint |
| Portability | Art.20 | Same JSON envelope as Art.15 + ML-readable structure | `GET /api/v1/patients/[id]/export` |
| Objection | Art.21 | Tracked as `objection` DSR kind; controller resolves | DSR endpoint |
| Not subject to automated decision-making | Art.22 | Engine output is **decision support**, not decision; no fully automated denial of care | Architectural |
| Withdraw consent | Art.7(3) | New row in `consent_records` with `granted=false` (immutable, versioned) | `POST /api/v1/consents` |

The DSR ledger (`data_subject_requests`) backs this with an auditable
state machine: `received → in_progress → fulfilled | rejected | cancelled`.
SLA: 30 days from `requested_at` (Art.12(3)). See
`api/v1/admin/dsr/[id]/process.ts`.

## 7. Pseudonymisation & anonymisation

- **Pseudonymisation in flight.** Internal joins use UUIDs only. Tenant
  identifiers are also UUIDs. There is no patient-identifying string in
  the URL path of audit-bearing endpoints (paths use UUIDs).
- **Anonymisation at rest (post-grace).** `fn_anonymize_patient()`
  (migration 003):
  - Sets identifying columns (`first_name`, `last_name`, `display_name`,
    `birth_date`, `external_code`, `contact_email`, `contact_phone`) to
    NULL or hashed placeholders.
  - Stamps `anonymized_at = NOW()`.
  - Drops PHI fields from `clinical_input_snapshot`, retaining only the
    score-relevant scalars + `engine_version` for clinical lineage.
  - Preserves computed scalars (scores, alerts, follow-up plans) so
    longitudinal cohort analytics remain possible without re-identifying
    the subject.
- Once `anonymized_at` is set, the row is no longer "personal data" under
  GDPR Recital 26 (subject to controller acceptance — `EXT-LEGAL`).

## 8. Hashing

- **IP hashing.** SHA-256 of the IP, applied at `withAuth` ingress. The
  raw IP is never stored in the database. The hash is non-reversible
  without an offline dictionary attack against the IPv4/IPv6 space (and
  even then carries only an actor-correlation signal — no name, no
  device).
- **User-agent.** Stored in truncated form (256 chars) to keep a
  fingerprint-like correlator without enabling per-pixel device
  fingerprinting.
- **No password hashes in app code.** Authentication delegates to Supabase
  Auth; passwords never traverse Uelfy code.

## 9. Cross-border data transfer

- Default deployment region: EU (Vercel + Supabase EU regions). `EXT-LEGAL`
  for non-EU deployments.
- No analytics SDK pings external endpoints from the clinical UI. The
  frontend bundle has zero third-party scripts beyond the Supabase SDK.
- The PDF rendering pipeline is fully in-process (`pdf-lib`); fonts are
  bundled (`scripts/fetch-noto-fonts.mjs`), not fetched at runtime, so no
  Google Fonts CDN traffic.

## 10. Children's data

The platform is designed for adult patients. There is no consumer-facing
sign-up for minors. If a tenant chooses to onboard paediatric patients,
the controller is responsible for (a) verifying parental consent under
Art.8 GDPR, and (b) configuring per-tenant retention windows
appropriately. `EXT-LEGAL`.

## 11. Sub-processors (template)

A current sub-processor list belongs in the per-tenant DPA (`EXT-LEGAL`).
At time of writing the platform's runtime sub-processors are:

- **Supabase** — Postgres + Storage + Auth host. EU region.
- **Vercel** — serverless function and edge host. EU region.
- **(Optional) Upstash Redis** — distributed rate-limit store, when the
  in-memory rate limiter is replaced.
- **(Optional) OpenAI** — bounded supportive AI commentary if a tenant
  enables it. **Disabled by default.** No PHI is sent in the prompt; only
  the bounded summary content. `EXT-LEGAL` to confirm with controller.

## 12. Privacy-impacting changes — change-control

Every schema change touching a PHI-bearing column or a retention boundary
is:

1. A numbered SQL migration in `supabase/migrations/`.
2. Mirrored by a code review that updates the relevant doc
   (`14-DELETION-POLICY.md`, this file, the GDPR matrix).
3. Captured in `11-CHANGELOG.md` under "Privacy".

A "no surprise" rule: if a change might alter what data is retained, for
how long, or who can see it, it is called out explicitly in the changelog
section header.

## 13. Open / EXT items

- `EXT-LEGAL` — Tenant-specific DPA template, sub-processor list, TIA for
  non-EU subjects.
- `EXT-LEGAL` — Lawful-basis assignment per data category per tenant.
- `EXT-LEGAL` — DPIA template (Art.35) — particularly required when
  onboarding paediatric or large-cohort tenants.
- Per-tenant retention overrides: schema columns exist on `tenants`
  (`retention_days_*`), the cron honours them; UI surface for
  tenant_admins to edit is partial (admin pages exist, full RBAC for
  tenant-scoped retention edits is on the roadmap).
- Optional encryption-at-rest envelope for `clinical_input_snapshot` with
  per-tenant KMS keys (current encryption is platform-managed AES-256 at
  the Supabase layer).
